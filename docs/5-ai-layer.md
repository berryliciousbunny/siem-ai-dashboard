## Overview

The AI layer is a Python FastAPI microservice that sits between the Wazuh Indexer and the SOC dashboard. Every alert generated by Wazuh passes through this service before reaching the analyst. The service runs three enrichment operations in sequence — anomaly scoring, user behaviour analysis, and plain-English summarisation — then writes the enriched alert back to OpenSearch where the dashboard picks it up.

The result is that every alert an analyst sees already carries a confidence score, a behavioural context flag, and a human-readable explanation — reducing the cognitive load of triage from minutes to seconds.

---

## Why AI on Top of Wazuh

Wazuh's rule engine is powerful but fundamentally reactive — it matches known patterns against a fixed ruleset. This works well for well-understood attack signatures but has structural limits that AI is well-positioned to address:

| Limitation of rules alone | How the AI layer addresses it |
|---|---|
| Only catches known attack patterns | Anomaly detection flags unknown/novel behaviour by learning what normal looks like |
| All alerts are equal weight | AI scores alerts 0.00–1.00 so analysts prioritise by confidence, not just rule severity |
| Rules don't adapt to your environment | UEBA builds a per-user baseline specific to your organisation — not generic thresholds |
| Alerts require manual interpretation | LLM generates a plain-English summary so L1 analysts don't need to decode every rule ID and log entry |
| No context across alert history | UEBA correlates current behaviour against a 30-day rolling history per user |

The AI layer doesn't replace Wazuh's detection engine — it amplifies it.

## Architecture

The microservice is structured as a sequential enrichment pipeline. Every new alert from OpenSearch passes through four modules in order:

```
OpenSearch (wazuh-alerts-*)
        │
        │  FastAPI poller checks every 30 seconds
        ▼
┌───────────────────────────────────────────┐
│           main.py — Alert Poller          │
│  Fetches new alerts from OpenSearch       │
│  Passes each alert through the pipeline   │
└───────────────────┬───────────────────────┘
                    │
                    ▼
┌───────────────────────────────────────────┐
│         anomaly.py — Anomaly Scorer       │
│  Isolation Forest model                   │
│  Input: alert features                    │
│  Output: anomaly_score (0.00–1.00)        │
└───────────────────┬───────────────────────┘
                    │
                    ▼
┌───────────────────────────────────────────┐
│         ueba.py — Behaviour Baseline      │
│  Per-user behavioural profiling           │
│  Input: user ID, source IP, timestamp     │
│  Output: ueba_flag (true/false) + reason  │
└───────────────────┬───────────────────────┘
                    │
                    ▼
┌───────────────────────────────────────────┐
│       summarizer.py — LLM Summariser      │
│  Sends alert to LLM API                   │
│  Input: full enriched alert JSON          │
│  Output: plain-English summary string     │
│  Also: VirusTotal hash/IP lookup          │
└───────────────────┬───────────────────────┘
                    │
                    ▼
        OpenSearch (wazuh-ai-enriched-*)
        Dashboard reads enriched alerts
```

## Module Breakdown

---

### `main.py` — FastAPI App & Alert Poller

**What it does:**
Entry point for the microservice. Runs a background polling loop that checks OpenSearch every 30 seconds for new alerts in `wazuh-alerts-*`. For each new alert, it runs the full enrichment pipeline and writes the result to `wazuh-ai-enriched-*`. Also exposes the REST API endpoints documented in the [API Reference](#api-reference) section.

**Key logic:**
```python
# Simplified polling loop
while True:
    new_alerts = opensearch.get_new_alerts(since=last_checked)
    for alert in new_alerts:
        alert["anomaly_score"] = anomaly.score(alert)
        alert["ueba"] = ueba.check(alert)
        alert["llm_summary"] = summarizer.summarise(alert)
        opensearch.write_enriched(alert)
    last_checked = now()
    time.sleep(30)
```

**Why FastAPI:**
FastAPI gives us automatic Swagger UI at `/docs`, async support for concurrent alert processing, and clean endpoint definitions with minimal boilerplate — well suited for a microservice that needs to be both a background worker and an API server simultaneously.

---

### `anomaly.py` — Isolation Forest Scorer

**What it does:**
Scores each incoming alert on a scale of 0.00 to 1.00, where scores closer to 1.0 indicate higher anomaly confidence. Scores are appended to the alert as `anomaly_score`.

**How Isolation Forest works:**
Isolation Forest is an unsupervised machine learning algorithm that detects anomalies by measuring how quickly a data point can be isolated from the rest of the dataset using random decision trees. Normal data points blend in with the crowd and take many splits to isolate. Anomalous data points are different enough that they get isolated in very few splits.

```
Normal alert (login at 9am from known IP)
  → Takes 47 random splits to isolate
  → Score: 0.12 (normal)

Anomalous alert (login at 3am from Tor exit node)
  → Isolated in 3 random splits
  → Score: 0.94 (high anomaly)
```

**Training data:**
The model is trained on 30 days of historical Wazuh alerts representing normal baseline activity. It does not need labelled attack data — it only needs to learn what normal looks like.

**Features used (input to model):**
```python
features = [
    "hour_of_day",           # 0–23
    "day_of_week",           # 0–6
    "source_ip_is_known",    # boolean — in known IP whitelist
    "alert_severity",        # Wazuh rule level 1–15
    "rule_id",               # encoded integer
    "bytes_transferred",     # outbound bytes if available
    "login_failure_count",   # failures in last 60s for this source
    "is_new_device",         # boolean — device seen before for this user
    "dest_port",             # destination port number
    "alert_count_last_1h"    # alert volume from this source in last hour
]
```

**Score interpretation:**

| Score | Interpretation | Recommended action |
|---|---|---|
| 0.00–0.30 | Normal — consistent with baseline | Monitor, low priority |
| 0.31–0.60 | Mildly anomalous — worth reviewing | L1 analyst review |
| 0.61–0.80 | Significantly anomalous | L1 triage, consider escalation |
| 0.81–1.00 | Highly anomalous — strong signal | Immediate L2 escalation |

---

### `ueba.py` — User & Entity Behaviour Analytics

**What it does:**
Builds a rolling 60-day behavioural profile for each user and flags alerts where the user's behaviour deviates significantly from their personal baseline. Appends `ueba_flag` (true/false) and `ueba_reason` (human-readable explanation) to each alert.

**How it works:**
```
Day 1–60: Learning phase
  → Tracks each user's: login hours, source IPs, devices,
    daily data volume, most common alert types
  → Stores rolling baseline per user_id in OpenSearch

Day 61+: Detection phase
  → New alert arrives for user john.doe
  → Compare current behaviour against john.doe's 60-day baseline
  → Login at 3am? → Outside normal window (08:00–18:00) → flag
  → Source IP 185.x.x.x? → Not in known IP list → flag
  → 4.5 GB transferred? → 22x daily average → flag
  → ueba_flag: true, ueba_reason: "Login time, IP, and data volume all outside baseline"
```

**Behavioural dimensions tracked per user:**

| Dimension | Baseline metric | Flag condition |
|---|---|---|
| Login time | Mean hour + std dev | > 2 std devs outside normal window |
| Source IP | Known IP whitelist | Any login from unseen IP |
| Device | Known device list | Any login from unseen device |
| Daily data volume | Mean bytes/day | > 3x daily average |
| Alert frequency | Mean alerts/day | > 5x daily average |

**Why per-user rather than global thresholds:**
A global rule like "flag logins after 7pm" will false-positive on night shift workers, remote teams in different time zones, and on-call engineers. UEBA avoids this by building a baseline specific to each individual — john.doe's normal is different from sarah.lee's normal, and the model respects that.

---

### `summarizer.py` — LLM Alert Summariser

**What it does:**
Sends the fully enriched alert JSON (including anomaly score and UEBA flag) to an LLM API and receives a plain-English summary. Also triggers VirusTotal lookups for file hashes and external IPs where applicable. The summary is appended to the alert as `llm_summary`.

**How it works:**
```python
# Build prompt from alert data
prompt = f"""
You are a SOC analyst assistant. Summarise the following security alert
in 3–5 sentences for a junior L1 analyst. Include: what happened,
why it is suspicious, which system is affected, and what the analyst
should do next. Be concise and avoid jargon where possible.

Alert data:
{json.dumps(alert, indent=2)}

Anomaly score: {alert['anomaly_score']} / 1.00
UEBA flag: {alert['ueba_flag']}
UEBA reason: {alert['ueba_reason']}
"""

response = llm_client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=300,
    messages=[{"role": "user", "content": prompt}]
)
```

**Example input → output:**

Input alert (simplified):
```json
{
  "rule_id": 100001,
  "rule_description": "Brute force attack detected",
  "source_ip": "45.33.22.11",
  "destination_host": "PROD-DB-01",
  "failure_count": 47,
  "timestamp": "2026-07-22T02:14:33Z",
  "anomaly_score": 0.91,
  "ueba_flag": true,
  "ueba_reason": "Source IP not seen before, activity at 2am outside baseline"
}
```

Output LLM summary:
```
⚠️ Brute Force Attack — High Confidence (AI score: 0.91)

47 failed SSH login attempts were made against production database server
PROD-DB-01 from external IP 45.33.22.11 between 02:14–02:15 AM. This IP
has never been seen before in this environment and the activity occurred
well outside normal hours — both flagged by the behaviour analysis engine.
This pattern is strongly consistent with an automated credential stuffing
attack. Recommended action: block source IP immediately, review all
successful logins from PROD-DB-01 in the past 24 hours, and escalate
to L2 if any successful authentications are found from this IP.
MITRE: T1110 | Severity: High | Rule: 100001
```

**LLM provider options:**
The provider is configurable via `.env` — no code changes needed to switch:

```env
# Option 1 — Anthropic Claude (paid, best quality)
LLM_PROVIDER=anthropic
ANTHROPIC_API_KEY=your_key_here

# Option 2 — OpenAI (paid, good quality)
LLM_PROVIDER=openai
OPENAI_API_KEY=your_key_here

# Option 3 — Ollama local model (free, no data leaves your network)
LLM_PROVIDER=ollama
OLLAMA_MODEL=mistral
OLLAMA_URL=http://localhost:11434

# Option 4 — Disable LLM (structured text summary only, fully free)
LLM_PROVIDER=none
```

## Model Details

### Isolation Forest — Training Process

```
1. Collect 30 days of Wazuh alerts from wazuh-alerts-* index
2. Extract the 10 features listed above from each alert
3. Normalise numerical features (StandardScaler)
4. Encode categorical features (LabelEncoder for rule_id)
5. Train Isolation Forest:
     n_estimators=100    (number of trees)
     contamination=0.05  (assume 5% of training data may be anomalous)
     random_state=42
6. Save trained model to ai/models/isolation_forest.pkl
7. Load model on microservice startup
8. Score each new alert in real time
```

### UEBA — Baseline Storage

User baselines are stored as JSON documents in OpenSearch under the `ueba-baselines` index — one document per user, updated daily:

```json
{
  "user_id": "john.doe",
  "baseline_updated": "2026-07-22",
  "login_hours": { "mean": 9.2, "std": 1.4 },
  "known_ips": ["10.0.1.45", "10.0.1.102"],
  "known_devices": ["WORKSTATION-07"],
  "daily_bytes_mean": 204800,
  "daily_bytes_std": 51200,
  "daily_alerts_mean": 2.1
}
```

## API Reference

The microservice exposes the following REST endpoints. Interactive docs available at `http://localhost:8000/docs`.

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/health` | Service health check — returns status and model load state |
| `GET` | `/alerts/enriched` | Fetch latest enriched alerts from OpenSearch |
| `POST` | `/alerts/score` | Manually submit an alert JSON for scoring |
| `GET` | `/ueba/baseline/{user_id}` | View the stored baseline for a specific user |
| `DELETE` | `/ueba/baseline/{user_id}` | Reset a user's baseline (e.g. after role change) |
| `GET` | `/model/info` | Returns model metadata — training date, feature list, version |
| `POST` | `/model/retrain` | Manually trigger model retraining on latest alert data |

## Limitations, Root Causes & Improvement Path

Being transparent about current limitations is important for honest evaluation of the PoC. Each limitation below includes why it exists and a concrete path to address it in future versions.

---

### 1. Cold Start Problem (UEBA)

**What it is:**
UEBA requires at least 30–60 days of historical data before the per-user baselines are meaningful. In the first weeks of deployment, the behaviour engine has little to compare against, so its flags are unreliable and should be treated with lower confidence.

**Why it exists:**
Statistical baselines need sufficient data points to calculate a reliable mean and standard deviation. With fewer than ~20 data points per user, the baseline is too noisy to be actionable.

**How to navigate it now:**
- Weight UEBA flags lower in the first 30 days — rely more on anomaly score and rule severity
- Display a "baseline confidence" indicator on the dashboard showing how mature each user's profile is
- Seed the baseline with historical logs from your existing SIEM or log archive if available

**Improvement path (v2.0):**
- Implement a "confidence tier" system — Tier 1 (< 7 days, low confidence), Tier 2 (7–30 days, medium), Tier 3 (30+ days, high) — and surface this in the dashboard
- Allow manual baseline seeding via the `/ueba/baseline/{user_id}` API endpoint using imported historical data

---

### 2. High Dimensional Data (Isolation Forest)

**What it is:**
Isolation Forest performance degrades when too many features (dimensions) are fed into the model simultaneously. This is known as the curse of dimensionality — with too many features, the algorithm struggles to find meaningful separation between normal and anomalous data points.

**Why it exists:**
Wazuh alerts contain 50+ fields. Feeding all of them to the model introduces noise that dilutes the signal from the features that actually matter for anomaly detection.

**How to navigate it now:**
Feature selection — we manually choose the 10 most informative fields (login hour, source IP novelty, bytes transferred, etc.) rather than feeding the full alert. This keeps the model performant and explainable.

**Improvement path (v2.0):**
- Add **PCA (Principal Component Analysis)** as a preprocessing step — mathematically compresses many features into fewer dimensions while preserving the most important variance
- Experiment with **Autoencoders** (a neural network approach) — better suited to high-dimensional data than Isolation Forest and can be trained to reconstruct normal alerts; anything it reconstructs poorly is anomalous
- Use **SHAP values** to explain which features drove each anomaly score — helps analysts understand why an alert was scored high

---

### 3. LLM API Cost at Scale

**What it is:**
Using Claude or OpenAI API for alert summarisation costs money — approximately $0.001–$0.01 per summary depending on alert length and the model used. At low PoC volumes (tens of alerts per day) this is negligible. At production SOC scale (thousands of alerts per day) costs can accumulate.

**Why it exists:**
Commercial LLM APIs charge per token. A typical alert summary uses ~500–800 tokens input + ~200 tokens output.

**How to navigate it now:**
- Use `LLM_PROVIDER=ollama` with a local open-source model (Mistral 7B or LLaMA 3) — runs on Laptop B, completely free, no data leaves the network
- Only summarise alerts above a certain anomaly threshold (e.g. only score ≥ 0.65 gets a summary) — reduces API calls by ~70%
- Cache summaries for duplicate or near-duplicate alerts — many brute force alerts look identical and don't need individual summarisation

**Improvement path (v2.0):**
- Implement **smart summarisation routing** — low-severity alerts get a rule-based template summary (free), high-severity alerts get full LLM summarisation (paid/local)
- Fine-tune a small open-source model specifically on security alert summarisation — better quality than generic Mistral at a fraction of the compute cost
- Add a **daily token budget** config variable — microservice stops calling LLM API once budget is exhausted and falls back to template summaries

---

### 4. No Feedback Loop (Model Stays Static)

**What it is:**
The Isolation Forest model is trained once on 30 days of baseline data and then stays fixed. It does not learn from analyst actions — if an analyst marks a low-scored alert as a real threat, the model doesn't update to score similar alerts higher in future.

**Why it exists:**
Building a proper feedback loop requires: capturing analyst actions, labelling alert data, managing a retraining pipeline, model versioning, and rollback capability — significant engineering complexity beyond the PoC scope.

**How to navigate it now:**
Analyst actions (dismiss as FP, escalate to L2, confirm as TP) are logged to OpenSearch under an `analyst-feedback-*` index — even though nothing acts on them yet. This means the labelled dataset is being built passively from day one, ready for a retraining pipeline when the time comes.

**Improvement path (v2.0):**
- Build a **weekly retraining pipeline** — scheduled job pulls the last 7 days of analyst-confirmed true positives and false positives from `analyst-feedback-*`, retrains the Isolation Forest with updated labels, and replaces the model if performance improves
- Implement **model versioning** — keep the last 3 model versions so you can roll back if a retrain degrades performance
- Add an **online learning variant** — models like Half-Space Trees can update incrementally in real time without full retraining, much more responsive to drift

---

### 5. Single-Node OpenSearch (No HA)

**What it is:**
The PoC runs OpenSearch as a single node. If the node goes down, alert storage and the dashboard go down with it — no redundancy.

**Why it exists:**
Single-node is sufficient for a PoC on one or two machines. A proper cluster requires at least 3 nodes and significantly more hardware.

**How to navigate it now:**
For a PoC this is acceptable — data loss risk is low in a test environment. Enable OpenSearch snapshots to back up alert data periodically.

**Improvement path (v1.0 → production):**
- Deploy a **3-node OpenSearch cluster** for production — one master node, two data nodes
- Use **Terraform scripts** (future roadmap item) to provision a production-grade cluster on AWS or Azure with automatic failover
